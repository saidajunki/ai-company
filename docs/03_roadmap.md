# 進め方（Roadmap / Tasks）

「まだ実装しない」前提で、まずは設計と運用ルールを固めるためのタスク。

## マイルストーン0: 憲法と台帳の確定（設計）

1. 会社憲法を確定（公開方針、承認範囲、$10/60分、WIP制限）
2. 10分レポートのフォーマットを確定（固定テンプレ）
2.1. 承認プロトコルを確定（Slackリアクション、request_id運用）
3. コスト台帳の最小スキーマを確定（トークン・概算）
4. 台帳の永続先（暫定）を確定（例: `companies/<company_id>/ledger/events.ndjson`）
5. 「常駐」の技術的実現方式を1つに決める（例: 会社コンテナ内で`manager`プロセスを常駐させる）
6. 憲法の変更プロセスを確定（AIは提案まで、適用はCreator承認が必要、等）
7. 意思決定ログのフォーマットを確定

受け入れ条件:
- 「何を記録し、何を公開し、何を承認に回すか」が迷わない文章になっている

## マイルストーン1: レジストリの確定（設計）

1. agent_registry（役割、権限、モデル、予算上限）の最小項目を確定
2. service_registry（作ったものの一覧）の最小項目を確定
3. “重複着手の検知”のやり方を決める（名前/ID/目的で照合）

受け入れ条件:
- 「誰が何を担当し、何が存在しているか」が一覧で追える

## マイルストーン2: 情報収集の方針確定（設計）

1. 外部情報のメタデータ（published_at/retrieved_at/source/query）を必須化
2. “追加で調べる判断”の基準を文章化（カテゴリ固定ではなく、状況依存）
3. 追加する情報源候補を列挙（無料→承認が必要なものの順）

受け入れ条件:
- 古い情報を新しい情報として扱わない（説明が日付ベースになっている）

## マイルストーン2.5: ドライラン（設計の穴出し）

実装に入る前に、手動でもよいので「運用が成立するか」を一度なぞって確認する。

例:
- 10分レポートのテンプレを使って、想定レポートを1回作る（WIP/差分/次/ブロッカー/コスト/配分）
- コスト台帳にダミーで1行書く（例: `companies/<company_id>/ledger/events.ndjson` に追記）
- 「20分レポートが来ない」時の叩き起こし手順を文章で通す（誰が/何を/どこへ通知）
- ドライラン結果を踏まえて、マイルストーン3（実装計画）をタスク分解する

受け入れ条件:
- 「で、このAIは何から始めるの？」に迷わず答えられる
- レポート・台帳・叩き起こしが文章上で一貫している
